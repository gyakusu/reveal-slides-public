SharePoint × Obsidian ではじめる社内 Wiki

対象: SharePoint を業務で使っているエンジニア・技術者

既存ツールの課題 — "ありがち" な困りごと

ドキュメントは書いている。でも「あの情報」が見つからない。

  • 「Loopに書いたはず」→ 検索しても出てこない
  • 「去年のトラブル対応、誰かまとめてたよな」→ バラバラで全体像が掴めない
  • 「オンボーディング資料を作ろう」→ 既存ドキュメントを探すたびに挫折

問題の本質:
情報が「ない」のではなく、
情報が「点在している」。
書いた本人すら再発見できない
「知識のブラックホール」状態。

書き方そのものを変える必要がある。
バラバラなドキュメントを「繋げる」仕組みが必要。

なぜ Microsoft Loop ではダメか

Loop はコラボツールとして優秀。でも「知識の蓄積」には構造的に向いていない。

課題 詳細
タグ付けができない ラベルによる横断的な分類・絞り込みが不可能
動的一覧化ができない 「ドラフト一覧」「領域別の公開済み記事」をクエリで作れない
グラフビューがない ドキュメント同士の繋がりを可視化する手段がない
アップデートが乏しい この1年、目立った機能追加がほとんどない

誤解なく: Loop をやめるのではない。
フロー型（議事録・リアルタイムコラボ）は Loop のまま。
ストック型の知識資産を置く場所として別の選択肢が必要。

目指す姿 — 有機的に繋がる資料

3つの価値をバランスよく取りに行く。

(a) 知見の再発見

リンクとグラフビューで「あれとこれが繋がっていた」という発見が起きる。過去の自分が半年後の自分を助ける状態。

(b) 暗黙知の構造化

口頭で伝わっているだけの知識を、Markdown とフロントマターで半構造化データにする。タグ・ステータスで再利用可能に。

(c) AI/LLM との相性

Markdown はプレーンテキスト。将来 RAG で社内AIアシスタントを作るとき、最も食わせやすい形式。今書いた資産が未来のAI資産になる。

なぜ Markdown か

プレーンテキストが一番強い。

  • 再利用性が高い — テキストエディタで開ける。grep で検索、git diff で差分
  • ツールロックインからの解放 — Obsidian をやめても .md ファイルはそのまま持っていける
  • Git との親和性 — バージョン管理が効く。SharePoint のバージョン履歴との二重安全網
  • LLM との親和性 — 現在の LLM が最も得意とする入力形式の1つ

ここから How の話に入ります。

ここまで（Why）が 5枚。
残り 14 枚は全部「どうやるか」です。

解決策アーキテクチャ — SharePoint × Obsidian

SharePoint はバックエンド、Obsidian はフロントエンド。

SharePoint
クラウド正本・同期・バックアップ・権限管理・バージョン履歴

OneDrive
SharePoint → ローカル PC への橋渡し。エクスプローラから直接操作可

Obsidian

セットアップ概要 — たった 3 ステップ

所要時間は 5 分。新しく構築するものは何もない。

① SharePoint ライブラリを選ぶ

Wiki として使うドキュメントライブラリを決める。既存のものを流用しても、新規作成でも OK。権限管理はここで設定。

② OneDrive 経由でエクスプローラ同期

ブラウザで SharePoint を開き「同期」ボタンを押す。Windows はエクスプローラ、Mac は Finder に Vault が出現。

③ Vault として開く

Obsidian 起動 → 「Open folder as vault」で同期先フォルダを指定。これだけで完了。

注意点: .obsidian フォルダの扱い
Obsidian の設定ファイル群。チームで Bases 定義・プラグイン設定を揃えるなら SharePoint 同期対象に含める（推奨）。

サンプル Vault のファイル構造

PARA（Projects / Areas / Resources / Archive）ベースの構成例。

Vault/
├── 00_inbox/         # とりあえず放り込む場所
├── 10_projects/      # 進行中のプロジェクト
│   ├── proj-alpha/
│   └── proj-beta/
├── 20_areas/         # 継続的に面倒を見る領域
│   ├── infra/
│   ├── security/
│   └── team-ops/
├── 30_resources/     # 参考資料・ナレッジ
│   ├── guides/
│   └── references/
├── 40_archive/       # 終わったもの
├── attachments/      # 画像・PDFなど
└── templates/        # Templater テンプレート

番号プレフィックスの狙い
エクスプローラ・サイドバーでの表示順を固定するため。アルファベット順だと archive が先頭に来て気持ち悪い。

ポイント: この構造は出発点。
運用しながらチームの実態に合わせて変えていく。
いきなり完璧を目指さない。

Obsidian の基本操作 — ライブデモ①

リンクで繋がり、双方向に見え、全体像がグラフで見える。

  • ノート作成 — サイドバー右クリック → 新規ノート
  • 内部リンク [[...]] — [[ と入力すると候補ポップアップ。存在しないノートでもリンクを先に書ける
  • バックリンク — 右サイドバーで「どこから参照されているか」を一覧表示
  • グラフビュー — Vault 全体のリンク関係をネットワーク図で可視化

メタデータ（フロントマター）の推奨スキーマ

Markdown ファイル先頭に YAML でメタデータを付ける。これが Bases の燃料になる。

---
title: ネットワーク障害対応手順 - DNS
aliases:
  - DNS障害対応
  - フェイルオーバー手順
tags:
  - area/infra
  - type/howto
  - status/published
created: 2026-03-15
updated: 2026-04-10
author: kususaki
status: published
related:
  - "[[20_areas/infra/DNS構成]]"
---
フィールド用途
title表示用タイトル（ファイル名と分離）
aliases別名・略称。[[略称]] でも解決
tags階層タグで横断分類
statusdraft / review / published / archived
relatedBases クエリで引ける関連リンク

スキーマはチームで統一することが最重要。統一されていないと、ただのコメントになる。

タグ・命名規則ガイドライン

フォルダとタグの棲み分け

フォルダ = 物理的な居場所（1ノート1フォルダ）
タグ = 論理的な属性（1ノートに複数タグ可）

階層タグ命名ルール

#area/infra          # 領域: インフラ
#area/security       # 領域: セキュリティ
#type/howto          # 種別: 手順書
#type/adr            # 種別: アーキテクチャ決定記録
#type/postmortem     # 種別: 障害報告書
#status/draft        # 状態: ドラフト
#status/published    # 状態: 公開済み
#status/archived     # 状態: アーカイブ済み

ファイル名・ID 規則

YYMMDD-<kebab-case-slug>.md

# 例
260410-dns-failover-howto.md
260315-postmortem-api-outage.md
260220-adr-authn-strategy.md

なぜ英数字ファイル名か？
日本語ファイル名は SharePoint の URL エンコードで文字数制限を踏む。実運用での地雷なので英数字 kebab-case に統一。

トップレベルタグは area / type / status / project の4つに絞るのがおすすめ。

Dataview / Bases でクエリ一覧化 — 本日のハイライト ⭐

Vault の Markdown ファイルを、メタデータで動的に一覧化できる。SQL でテーブルを引く感覚。

ドラフト一覧（週次棚卸し用）

TABLE title AS "タイトル",
       author AS "作成者",
       updated AS "最終更新"
FROM ""
WHERE status = "draft"
SORT updated DESC

90日以上更新なしの記事を洗い出す

TABLE title, author, updated,
  (date(today) - date(updated)).days
    AS "経過日数"
FROM ""
WHERE status = "published"
  AND (date(today) - date(updated)).days > 90
SORT updated ASC

PDF のタグ付け — サイドカード方式

PDF の隣に .md を置けばタグ付けできる。プラグイン不要。

├── vendor-spec-apigw.md   ← サイドカード
├── contract-2026q1.pdf
└── contract-2026q1.md     ← サイドカード
---

tags:
  - type/spec
  - area/infra
  - vendor/acme
source: vendor-spec-apigw.pdf

  mTLS認証、10k rps想定。
  3章フェイルオーバーはDNS構成と関連。
---

![[vendor-spec-apigw.pdf]]

この方式のうまみ

  • プラグイン不要 — 標準 Obsidian で動く
  • 検索可能 — 要約・メモがVault内検索にヒット
  • Bases 対応 — type/spec の一覧クエリにPDFが混ざって出る
  • 埋め込み表示 — ![[...pdf]] でエディタ内にインライン表示

「PDF はタグ付けできない」という先入観を外そう。隣に .md を置けばタグ付けできる。

Obsidian 特有のコツ — ライブデモ③

4つの便利機能。まずはデイリーノートから始めるのがおすすめ。

1. Templater プラグイン
ノート作成時にテンプレートを自動適用。
<% tp.date.now("YYYY-MM-DD") %> でフロントマターの created を自動入力。

2. デイリーノート
日付ごとに1ノート自動生成（daily/2026-04-10.md）。
その日の考え・読んだPDF・会議メモを全部ここに放り込む。

3. 埋め込み ![[...]]
他ノートをインラインで埋め込む。部分埋め込み（![[note#見出し]]）も可。情報を重複させずに参照できる。

4. Outgoing Links パネル
いま開いているノートから出ていくリンク一覧。バックリンクと組み合わせて「ノートを関係の中心として捉える」視点を得る。

コンフリクト対策 — 1ファイル1トピックの原則

技術では解決しない。運用ルールで解決する。

OneDrive 同期 ≠ リアルタイム共同編集
同じ .md を複数人が同時編集するとコンフリクトコピーが発生する。

❌ 悪い例

infra-handbook.md
  （500行、全領域を網羅した巨大ファイル）

✅ 良い例

infra/
├── dns-setup.md
├── dns-failover.md
├── vpn-config.md
└── monitoring.md

3つのルール

  • 1ファイル1トピック — ファイルを細分化。同じファイルを複数人が同時に触る確率を下げる
  • フロー型は Loop に任せる — 議事録・リアルタイムコラボは Loop のまま。ストック型だけ Obsidian
  • SharePoint バージョン履歴を保険に — 最大500バージョン保持。万が一の復元に使う
📸 ファイル細分化前後の比較

検索の活用 — 二層の検索戦略

書く人は Obsidian、読むだけの人は SharePoint。両方が同じファイルを見ている。

第1層: Obsidian 内検索

Cmd+Shift+F（Mac）/ Ctrl+Shift+F（Win）

  • 全文検索（本文の文字列）
  • タグ検索: tag:#area/infra
  • メタデータ検索: [status:published]
  • 正規表現: /pattern/

第2層: SharePoint 側検索

Obsidian を持っていない人も使える

  • 経営層・他部署メンバーが「ちょっと調べたい」ときに使う
  • Microsoft 365 統合検索（Teams / Outlook と横串）
  • エクスプローラの検索ボックスからも可

セキュリティ・ガバナンス

SharePoint がバックエンドなので、ガバナンスは SharePoint に委譲できる。

権限管理は SharePoint 側で

  • Obsidian には権限の概念がない。Vault を開ける人はすべて見える
  • 部門ごとにライブラリを分ける
  • 機密情報は別ライブラリに隔離し、必要な人だけ同期
  • 1つの巨大Vaultにすべて突っ込まないのがポイント

Wiki に書いてはいけないもの:
個人情報・顧客データ・パスワード
→ パスワード管理ツールや Key Vault に置き、Wiki からはリンクだけ書く

退職者対応:
アカウントを消しても .md ファイルは SharePoint に残る。author フィールドで文脈が辿れる。ファイルはユーザーではなく Vault に帰属する（Loop では不可能）。

まとめ — 明日から始める 3 ステップ

今日の午後にでも始められる。所要時間は 15 分以内。

① Vault を立ち上げる

SharePoint ライブラリを1つ選び、「同期」ボタンを押す。同期先フォルダを Obsidian で「Open folder as vault」。5分で完了。

② 1ノート書く

今日の業務から何か1つ手順や知見をノートに。スライド10のフロントマターをコピペし、title / tags / status だけ埋める。#status/draft で OK。

③ Bases で体感する

スライド12のクエリ例1をそのまま貼り付ける。#status/draft のノートが一覧に出る。「これは Loop ではできなかった」を自分の手で味わう。

まずは一人で Vault を持ってみるのが最良のスタート。
チームに広める話はまた別の機会に。
