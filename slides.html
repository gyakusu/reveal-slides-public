Claude Code
スタートダッシュガイド

対象: LLMでコード相談の経験があり、Pythonが書ける新入社員
前提: Claude Codeについては知識ゼロからスタート

このガイドの思想

Claude Codeは強力だが、「道具としての性質」を理解しないと効果が出ない。核となる考え方は3つ。

① セッションを使い捨てろ

LLMのコンテキストは有限の資源。相談と実装を同じセッションで行わず、結論をファイルに書き出して新しいセッションで次に進む。

② ツールに任せろ

集計や統計処理はPythonスクリプトで。品質管理はmypyやruffで。LLMは確率的に動く。決定的な処理はツールに委ね、LLMはそれを「使う側」に徹する。

③ コンテキストを守れ

探索や分析はサブエージェントに委譲し、メインには結果だけを返させる。必要な知識は必要なときだけ読み込む。

3番目の「必要なときだけ読み込む」を、このガイドでは段階的開示と呼ぶ。
LLMにとっての段階的開示とは、目的に応じた動的コンテキスト読み込みのことだ。
CLAUDE.mdに何でも書くのではなく、特定タスクで必要になる知識はその瞬間にだけロードする。
この段階的開示を実現する最も有用な仕組みが Skills である。

インストールと最初の起動

インストール

# インストール（Node.js 18以上が必要）
npm install -g @anthropic-ai/claude-code

# プロジェクトのルートに移動して起動
cd your-project
claude

初回起動時にログインを求められる。それだけで準備完了。

確認コマンド

claude --version    # バージョン確認
claude --help       # ヘルプ表示

前提: Node.js 18以上
起動: プロジェクトのルートディレクトリで claude
終了: /exit または Ctrl+C

CLAUDE.md — プロジェクトの「取扱説明書」

CLAUDE.mdはプロジェクトルートに置くMarkdownファイルで、Claude Codeがセッション開始時に毎回読む。
人間にとっての「プロジェクトのREADME」に相当するが、読者がLLMである点が異なる。

書くべきこと

  • ビルド・テスト・リントの実行コマンド
  • プロジェクト構造の概要（モノレポならアプリとパッケージの対応）
  • コーディング規約のうち、ツールで強制できないもの（設計方針など）
  • 「やってはいけないこと」の明示

書いてはいけないこと

  • ツールで強制できるルール（フォーマットはprettier、型チェックはmypy）
  • コードスニペットのコピペ（すぐ陳腐化する。ファイルパスで参照せよ）
  • 何でもかんでも（指示が多すぎると全体的に無視されはじめる）

CLAUDE.md — 良い例

# Project Overview
Python 3.12 / FastAPI / PostgreSQL のバックエンドAPI。

# Commands
- テスト: pytest tests/ -x --tb=short
- 型チェック: mypy src/ --strict
- リント: ruff check src/
- 単一テスト: pytest tests/test_xxx.py::test_name -v

# Architecture
- src/api/ — ルーティング
- src/domain/ — ビジネスロジック（外部依存なし）
- src/infra/ — DB・外部API

# Rules
- domain層はinfra層をimportしてはならない
- 新しいエンドポイントには必ず対応するテストを書く
- エラーハンドリングはsrc/api/errors.pyのパターンに従う

CLAUDE.md — 重要な原則と配置場所

重要な原則

  • 短く保て。 各行について「これを消したらClaudeが間違えるか？」と自問する。NOなら消す。
  • ポインタを使え。 コードをコピペするのではなく src/api/errors.py を参照 と書く。
  • @構文でファイルを参照できる。 @README.md や @docs/architecture.md で外部ファイルを読み込める。
  • 定期的に手入れしろ。 Claudeが指示を無視し始めたら、CLAUDE.mdが肥大化しているサイン。

配置場所の使い分け

場所 用途
~/.claude/CLAUDE.md 全プロジェクト共通の個人設定
./CLAUDE.md プロジェクト固有（gitにcommitしてチームで共有）
./子ディレクトリ/CLAUDE.md サブモジュール固有（必要時のみ読み込まれる）

基本ワークフロー — セッションの使い捨て

最も大事な習慣: 1タスク1セッション。LLMのコンテキストは積み重なるほど品質が劣化する。

推奨フロー

[セッション1: 相談]
  「このAPIの設計方針を相談したい」
  → 議論の結論をファイルに書き出させる
  → セッション終了

[セッション2: 実装]
  「docs/plan.md に書かれた方針に従って
   APIを実装して」
  → 新鮮なコンテキストで実装に集中
  → セッション終了

ストレージへの逃がし方

# セッション内でClaudeに指示する例
「今の議論の結論を
 docs/design-decision-auth.mdに書き出して」

「実装計画を
 docs/plan-user-api.mdにまとめて」

次のセッションではそのファイルを読むだけで済む。
コンテキストウィンドウには結論だけが入り、議論の過程は入らない。

コンテキスト管理の小技

/compact

コンテキストを要約して圧縮する。長くなってきたら手動で実行する。

/clear

タスクを切り替えるときにリセット。コンテキストを完全にクリアして新鮮な状態で再開する。

/model haiku

機械的な作業（リネーム、フォーマット確認など）はHaikuで十分。コスト・速度ともに優位。

/effort low

深い推論が不要な作業に。レスポンス速度が上がり、コストも削減できる。

LLMにやらせるな — スクリプト化と冪等性

原則: 決定的な処理はスクリプトに書け。 LLMは確率的に動く。掛け算ですら間違える可能性がある。

悪い例

「このCSVの売上列を合計して」
→ LLMが自力で計算する
→ 間違える可能性あり

LLMは確率的に動く。
同じ入力でも毎回同じ結果とは限らない。

良い例

「このCSVの売上列を合計するPython
 スクリプトを書いて、
 scripts/sum_sales.py に保存して、
 実行して」
→ スクリプトが確定的に計算する
→ 再実行も可能

スクリプトは冪等。
何度実行しても同じ結果が返る。

品質管理もツールに委ねる

CLAUDE.mdに書いておくと効果的

- コード変更後は必ず
  `ruff check src/` と
  `mypy src/ --strict` を実行する
- テストは `pytest tests/ -x` で確認する
- これらが通らない状態で
  タスク完了とみなさない

さらに確実に — hooksで強制する

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "ruff check src/ && mypy src/ --strict"
          }
        ]
      }
    ]
  }
}

CLAUDE.mdの「お願い」 vs hooksの「強制」 — 確定的なゲートを作るにはhooksを使う。
Claudeがファイルを編集するたびに自動でリンターと型チェッカーが走る。

mypyのもう一つの価値 — 型はLLMにとってのドキュメント

型アノテーションの付いたコードは、LLMにとってコードとドキュメントが一体化したモノリシックな資料になる。

NG: 型なし — LLMは使い方を推測するしかない

def calculate_discount(price, user, campaign):
    if campaign and user.is_premium:
        return price * campaign.rate
    return price

priceはintかfloatか？campaignはNoneが来うるか？
LLMは推測するしかない。推測は確率的であり、間違える。

OK: 型あり — 関数シグネチャ自体がドキュメント

from decimal import Decimal

def calculate_discount(
    price: Decimal,
    user: PremiumUser | StandardUser,
    campaign: DiscountCampaign | None,
) -> Decimal:
    if campaign is not None and isinstance(user, PremiumUser):
        return price * campaign.rate
    return price

priceはDecimal・campaignはNone許容・戻り値はDecimal。
型は実行時（mypy）に検証されるため、常に正確なドキュメントになる。

サブエージェントへの委譲 — コンテキストを守る

ファイル探索やコード分析はコンテキストを大量に消費する。メインセッションでそれをやると、肝心の作業時にコンテキストが汚染される。

組み込みサブエージェント

Explore

読み取り専用のコードベース探索。ファイル構造やコード検索を、メインコンテキストを汚さず実行する。

Plan

計画モード時にコードベースを調査し、計画を立てる。実装前の設計フェーズで活用する。

汎用サブエージェント

探索と変更の両方が必要な複雑なタスク用。メインには結果だけが返る。

「Exploreサブエージェントを使って、認証関連のファイルを全部洗い出して」
「この変更の影響範囲をサブエージェントに調べさせて、結果だけ教えて」

カスタムサブエージェントの作り方

.claude/agents/ にMarkdownファイルを置く:

<!-- .claude/agents/code-reviewer.md -->
---
name: code-reviewer
description: コード変更のレビュー。
  差分を読んで問題点を報告する。
tools:
  - Read
  - Grep
  - Glob
model: sonnet
---

あなたはコードレビュアーです。
与えられた差分を読み、以下の観点でレビュー:
- バグの可能性
- 設計原則への違反
- テストの漏れ

出力形式:
- 問題の重要度（高/中/低）
- ファイルパスと該当箇所
- 具体的な修正提案

ポイント

  • tools で使えるツールを制限する
    （レビュアーに書き込み権限は不要）
  • model でコストを制御する
    （レビューならsonnetで十分なことが多い）
  • サブエージェントはネストできない（1階層のみ）
  • サブエージェントの結果だけがメインに返るので、コンテキストが節約される

いつ委譲するか

状況 やること
「このプロジェクトの構造を把握したい」 Exploreに委譲
「この変更の影響範囲を調べたい」 サブエージェントに委譲
「テスト結果を分析してほしい」 サブエージェントに委譲
「このファイルを修正して」 メインで直接やる
「設計方針を相談したい」 メインで直接やる（その後セッション切り替え）

Skills — 段階的開示の中核

CLAUDE.mdは毎回読み込まれる。Skillsは必要なときだけ読み込まれる。
「常に必要な情報 → CLAUDE.md」「特定タスクで必要な情報 → Skill」と使い分けることで、
コンテキストを目的に応じて動的に制御できる。これが段階的開示の実装だ。

skillの作り方

<!-- .claude/skills/deploy/SKILL.md -->
---
name: deploy
description: 本番環境へのデプロイ手順
---

# デプロイ手順

1. mainブランチが最新であることを確認
2. scripts/pre-deploy-check.sh を実行
3. scripts/deploy.sh production を実行
4. scripts/post-deploy-verify.sh でヘルスチェック

skillの設計方針

  • skillの中にスクリプトやテンプレートを同梱できる（.claude/skills/deploy/scripts/ など）
  • Claudeにボイラープレートを「生成」させず、同梱ファイルを「使わせる」ことで再現性を上げる
  • skillの作り方がわからなければ、Claudeに聞く:
    「デプロイ用のskillを作りたい」

CLAUDE.md vs Skill:
常駐する知識 → CLAUDE.md
タスク固有の知識 → Skill

設定ファイルのベストプラクティス

.claude/settings.json の構成例

{
  "permissions": {
    "allow": [
      "Read",
      "Grep",
      "Glob",
      "pytest tests/**",
      "ruff check src/",
      "mypy src/ --strict"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "ruff check src/"
          }
        ]
      }
    ]
  }
}

設定の方針

  • permissions.allow で頻出コマンドの毎回承認を省略する
  • hooksでリンター・フォーマッターを自動実行する
  • CLAUDE.mdの「お願い」よりhooksの「強制」を優先する（確定的 > 確率的）
  • .claude/settings.json はgitにcommitしてチームで共有する

/init で始める

cd your-project
claude
# セッション内で
/init

/init はプロジェクト構造を分析してCLAUDE.mdの雛形を生成する。ここから不要な記述を削って洗練させていく。

カスタムコマンド — 繰り返し作業のパッケージ化

.claude/commands/ にMarkdownファイルを置くと、スラッシュコマンドとして使える。

<!-- .claude/commands/review-pr.md -->
---
description: PRの差分をレビューする
model: sonnet
---

git diffの結果をレビューしてください。
対象ブランチ: $ARGUMENTS

観点:
- バグの可能性
- テストの漏れ
- CLAUDE.mdのルールへの違反

問題がなければ「LGTM」とだけ返してください。

使い方

/review-pr feature/user-auth
  • $ARGUMENTS にコマンドの引数が入る
  • チームで共有するコマンドはプロジェクトの .claude/commands/ に配置
  • 個人用は ~/.claude/commands/ に配置

繰り返し使う作業手順をコマンド化することで、
チーム全員が同じ品質でタスクを実行できる。

よく使うコマンド一覧

コマンド 用途
/initCLAUDE.mdの雛形を生成
/compactコンテキストを要約・圧縮
/clearコンテキストをリセット
/model haiku軽量モデルに切り替え
/model opus高性能モデルに切り替え
/effort low推論の深さを下げる
/plan計画モード（実装前に計画を立てる）
/status現在の設定状態を確認
/doctor環境の問題を診断
/resume過去のセッションを再開
/hooks設定済みhookの一覧
/agentsサブエージェントの管理

ワンライナーでの利用

# パイプで渡す
cat error.log | claude -p "このエラーの原因を特定して"

# 非対話で実行
claude -p "tests/ 以下のテストを全部実行して結果を報告して"

-p フラグ: プロンプトを直接渡して非対話で実行。
シェルスクリプトや CI/CD パイプラインへの組み込みに便利。

まとめ — 明日から使う3つの習慣

① セッションを使い捨てろ

相談したら結論をファイルに書き出して、新しいセッションで実装に入る。

② ツールに任せろ

品質管理はhooksで自動化、計算はスクリプトで。LLMは「使う側」。

③ コンテキストを守れ

探索や分析はサブエージェントに委譲し、メインには結果だけ返させる。タスク固有の知識はSkillsで必要なときだけ読み込む。

これらを意識するだけで、Claude Codeの効果は大きく変わる。
慣れてきたら、カスタムコマンド、agent teams、git worktreeなど、さらに高度な機能に進んでいこう。
