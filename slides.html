なぜセキュリティが重要か — AIエージェントの二面性

AIエージェントは高い生産性をもたらすが、新たなリスクベクターも導入する。

① 広範なコンテキスト

高パフォーマンスにはAPIキー、環境変数、独自ロジックへのアクセスが必要。

② ゼロトラストの姿勢

AIエージェントを「強力だが完全には信頼できないエンティティ」として扱う。

③ 多層防御が不可欠

シークレット拡散、シェル履歴、プロンプトインジェクション等の多層的リスクに対応。

2025年だけで2,800万件以上のハードコードされたシークレットが公開リポジトリに追加（前年比34%増）。
漏洩したAWSキーは数分以内に自動クローラーでテストされる。

脅威カテゴリの全体像

脅威カテゴリ 露出のメカニズム 潜在的影響
シークレットの拡散 .env、.json、YAML設定ファイルへのハードコード クラウドインフラ全体の侵害
シェル履歴の脆弱性 コマンドライン引数として渡されたAPIキーのログ記録 .bash_historyを通じた永続的な資格情報アクセス
エージェントの越権行為 サブエージェントによるローカル設定の拒否ルールのバイパス SSHキーやGPG構成情報の外部送信
プロンプト注入 ソースコードやREADMEに埋め込まれた悪意ある指示 rm -rfやcurlによるデータ抽出などの不正コマンド実行

Claude Codeの内部セキュリティ・アーキテクチャ

「ローカル優先」設計 — ソースコードはローカル環境内に留まる。

パーミッションベースのアーキテクチャ

  • ファイル編集、bash実行、ネットワークリクエストに明示的な承認が必要
  • 書き込みは起動ディレクトリとサブディレクトリに限定
  • プロジェクト外は読み取りのみ可能

残存するリスク

  • プロジェクト外の「読み取り」権限は依然リスク
  • 曖昧な指示で ~/.ssh や ~/.aws を「探索」する可能性
  • TLS暗号化されたAPI通信にコンテキストが含まれる

セキュリティ構成の責任は完全にユーザーにある。

Gitエコシステムとの統合リスク

自動読み取りの問題

セッション開始時にリポジトリのステータスやコミット履歴を含むGitメタデータを自動的に読み取る。

  • 大規模プロジェクトではGitログだけで数万トークンを消費
  • トークンの浪費がコスト・品質に影響

「追記専用」データモデルの危険性

過去に一度でもコミットされたシークレットは、HEADから削除されていても履歴からアクセス可能。

  • AIエージェントはGit履歴を遡ってシークレットを発見しうる
  • リポジトリレベルのシークレットスキャンと履歴のクレンジングが不可欠

階層的な設定管理による防御戦略

設定は4つの階層で適用される。上位が優先。

設定スコープ Ubuntu/Linux上の場所 優先順位 目的
Managed (管理) /etc/claude-code/managed-settings.json 1 (最高) 組織全体で強制。ユーザー上書き不可
Local (ローカル) .claude/settings.local.json 2 個別開発者のプロジェクト固有設定
Project (プロジェクト) .claude/settings.json 3 チーム共有設定。Gitで管理
User (ユーザー) ~/.claude/settings.json 4 (最低) 全プロジェクト共通のグローバル設定

permissions.deny — 機密ファイルへのアクセス遮断

セキュリティ構成の核心。~/.ssh/ や ~/.aws/ へのアクセスを明示的に拒否。

推奨される設定例

{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(~/.ssh/**)",
      "Read(~/.aws/**)",
      "Bash(curl:*)",
      "Bash(wget:*)",
      "Bash(cat .env:*)",
      "Bash(cat ~/.ssh/*:*)",
      "Bash(cat ~/.aws/*:*)"
    ]
  }
}

重要な注意点

  • Read(./.env) はClaude標準ツールのみ制限
  • cat .env 等のBashは別途denyが必要
  • curl や wget も拒否リストに登録すべき

注意: 標準ツールのdenyだけでは不十分。Bash経由のアクセスも必ずブロックする。

資格情報管理① — direnvによるディレクトリ単位の制御

.envファイルへの直接保存を避け、ディレクトリに応じた環境変数の自動ロード/アンロードを実現。

仕組み

# .envrc（プロジェクトルート）
export AWS_ACCESS_KEY_ID="AKIA..."
export ANTHROPIC_API_KEY="sk-ant-..."

# ディレクトリ移動時
cd your-project   # → 変数が自動ロード
cd ..             # → 変数が自動アンロード

ディレクトリを抜けると変数は自動的に消える。

セキュリティポイント

  • .envrc は決してGitにコミットしない
  • グローバルな .gitignore で全プロジェクトから除外
  • 変更時は direnv allow で明示的承認が必要
  • 別ディレクトリのAIセッションが機密情報を継承しない

資格情報管理② — secret-toolとGNOMEキーリング

資格情報をプレーンテキストから完全に排除し、OSのキーリングに保存する。

保存と取得

# シークレットの保存
secret-tool store --label="Anthropic API Key" \
  service anthropic user developer

# .envrcから取得（direnvと組み合わせ）
export ANTHROPIC_API_KEY=$(
  secret-tool lookup service anthropic \
    user developer
)

メリット

  • APIキーはメモリ上にのみ存在
  • ディスク上のファイルには書き込まれない
  • シェル履歴にキーが残るリスクを劇的に低減
  • libsecretライブラリのCLIインターフェース

direnv + secret-tool の組み合わせが最も安全なパターン。

自動化された防御策 — シークレットスキャンと検証

手動設定だけでは不十分。多層防御の一環として自動スキャンツールを開発サイクルに組み込む。

Gitleaks

軽量なプリコミットフック。高エントロピー文字列やAWS/Anthropicキーパターンを検出し、commitを未然にブロック。

TruffleHog

パターンマッチングを超えた「検証」機能。APIと通信してキーが実際に有効か確認。--only-verified で誤検知を排除。

OIDC認証

CI/CDでOpenID Connectを採用し、静的キーを完全排除。GitHub Actionsから一時的な短命資格情報を取得。

Gitleaks（ローカル）→ TruffleHog（CI/CD）→ OIDC（究極解） の順で防御を強化。

プロンプトインジェクションとサブエージェント対策

自律モードやサブエージェントの普及に伴い、攻撃表面積は拡大。
AIレベルのパーミッションを過信せず、OSレベルのサンドボックスを構築する。

セキュリティ対策 実装方法 防御効果
サンドボックス化 /sandbox コマンド、Ubuntu上の bubblewrap サブエージェント含む全プロセスの不正アクセス防止
フックの無効化 "disableAllHooks": true 自動実行スクリプトによる悪意あるコードの永続化防止
MCPサーバーの制限 信頼できるサーバーのみ許可リスト登録 悪意あるModel Context Protocolサーバーの実行防止
ソフトデナイ autoMode.soft_deny PII・本番環境データへの接触をAIレベルで警告

まとめ — 多層防御ロードマップ

① シークレット排除

direnv + secret-toolでプレーンテキストを排除。

② アクセス制御

permissions.denyの階層的適用でOS機密領域を遮断。

③ 自動スキャン

Gitleaks + TruffleHogでコミット前に最終防波堤。

④ 環境隔離

bubblewrap / DockerでLinuxカーネルレベルの防御。

AIエージェントを「全権限を持つ可能性があるが、完全には信頼できない実習生」として扱う。
ローカル環境のセキュリティ確保は、現代のソフトウェアエンジニアリングにおける不可欠なスキルである。
