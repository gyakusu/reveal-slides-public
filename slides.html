Claude Codeの基本的なアクセス権限

まず「Claude Codeに何ができるのか」を正確に理解しよう。

操作 範囲 補足
書き込み 作業ディレクトリとそのサブディレクトリのみ プロジェクト外への書き込みは不可
読み取り 制限なし（プロジェクト外も読める） ~/.ssh や ~/.aws も読める
Bash実行 初回はユーザー承認が必要 一度許可すると広範に実行可能

大原則：自分が読めるものは、Claude Codeも全て読める。
Claude Codeは実行ユーザーの権限で動作する。特別な権限昇格はないが、制限もない。

Q1. 見せたくないファイルを守るには？

例：Pythonで解析が必要だが、Claude Codeには中身を見せたくないCSVがある。

方法①: permissions.denyでブロック

{
  "permissions": {
    "deny": [
      "Read(data/sensitive/*.csv)",
      "Bash(cat *sensitive*)",
      "Bash(head *sensitive*)"
    ]
  }
}

Claude Codeがファイルを直接読めなくなる。スクリプト作成の支援は引き続き可能。

方法②: コードだけ書かせて実行は自分で

カラム名などの構造情報だけClaude Codeに伝えてスクリプトを作成してもらい、実行は別ターミナルで手動で行う。最もシンプルかつ確実。

なぜ方法②が重要か：deny設定でファイルの直接読み取りを防いでも、Claudeがスクリプトを実行すると標準出力経由で機密データを見てしまう（例：df.head()の結果）。実行を分離すればこの間接リークを完全に防げる。

状況別おすすめ

手軽にやりたい deny設定 + 実行は自分で
組織でルール化 deny + ディレクトリ分離
機密性が非常に高い サンドボックス（後述）

Q2. permissions.denyは迂回されない？

結論：設定の仕方次第では迂回されうる。

迂回の具体例

Read(~/.ssh/*) をdenyしても…

# これらはBashツール経由なので別途ブロックが必要
cat ~/.ssh/id_rsa
python3 -c "print(open('/home/user/.ssh/id_rsa').read())"

ReadツールのdenyはClaude Codeの組み込みツールだけを制限する。Bash経由のアクセスは別途denyが必要。

さらに厄介な迂回：出力経由の間接リーク

# Claude自身が書いたスクリプトをBashで実行すると…
import pandas as pd
df = pd.read_csv("data/sensitive/customers.csv")
print(df.columns)  # → Claudeがこの出力を読んでしまう
print(df.head())   # → 機密データがClaudeに漏れる

ファイルの直接読み取りをdenyしても、スクリプト経由の標準出力はClaudeに返される。

対策の比較

対策 迂回リスク 手軽さ
permissions.deny パターン漏れの可能性あり 簡単
サンドボックス 非常に低い（カーネルレベル） 簡単
OS専用ユーザー 非常に低い やや手間
Docker隔離 ほぼゼロ 初期構築に手間
Hooks + CLAUDE.md 間接リークを軽減 簡単
実行を分離（方法②） 間接リークをゼロに やや手間

permissions.denyは第一の防御線。間接リーク対策にはHooks・CLAUDE.md・実行分離を組み合わせよう。

設定ファイルの階層と場所

Claude Codeの設定は4つの階層がある。上位の設定が優先される。

設定スコープ ファイルの場所 優先順位 用途
Managed（管理） /etc/claude-code/managed-settings.json 1（最高） 組織全体のポリシー。ユーザーが上書き不可
Local（ローカル） .claude/settings.local.json 2 個人のプロジェクト固有設定（gitignore推奨）
Project（プロジェクト） .claude/settings.json 3 チーム共有設定。Gitで管理
User（ユーザー） ~/.claude/settings.json 4（最低） 全プロジェクト共通のグローバル設定

新入社員へのおすすめ：まずは ~/.claude/settings.json（User設定）に
permissions.denyを書いておけば、全プロジェクトに適用される。

Q3. サンドボックスって何？どう使うの？

有効化の方法

# 必要なパッケージをインストール
sudo apt-get install bubblewrap socat

# Claude Code内でメニューから有効化
# → /sandbox コマンドで設定メニューを開く
# または settings.json で設定

制限される範囲

  • ファイルシステム：書き込みは作業ディレクトリのみ
  • ネットワーク：全外向き接続がプロキシ経由、許可ドメインのみ通信可

サンドボックスの対象範囲

サンドボックスは主にBashツール経由の実行をOSレベルで封じ込めるもの。

注意：Read / Edit / Write などClaude Codeの組み込みツールはサンドボックスを経由しない（permissions設定で制御）。

最も効果的な組み合わせ：
permissions.deny（組み込みツール制御）
+ サンドボックス（Bash制御）

Q3.5 出力経由の間接リークを防ぐには？

サンドボックスやdenyで直接アクセスを防いでも、スクリプトの実行結果（stdout）経由でClaudeが機密データを見てしまう問題がある。

対策①: PreToolUse Hooksで実行前検閲

// settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/block-sensitive-output.sh"
          }
        ]
      }
    ]
  }
}

フックスクリプトで df.head() や print(df) 等のパターンを含むコマンドを実行前にブロックできる。

対策②: CLAUDE.mdでルール化

# CLAUDE.md に記載
## データ分析ルール
- 分析結果はoutput/に
  ファイル出力すること
- print()で機密データを
  標準出力に出さないこと
- 結果確認は
  「output/result.csvに書き出しました」
  と報告するだけにすること

注意：CLAUDE.mdはプロンプトレベルの防御。通常利用には実用的だが、完全な保証はない。

機密性が高い場合の鉄則：
コードだけ書かせて実行は別ターミナルで行う（方法②）。これが唯一の完全な対策。

Q4. 他の人のファイルやOS領域は読まれない？

Claude Codeは実行ユーザーの権限で動作する。Linuxのファイルパーミッションがそのまま適用される。

アクセス可否の具体例

対象 読める？ 理由
/home/friend/ 不可 OSパーミッション（750）で拒否
~/.ssh/ 可能 自分のファイルだから
/etc/passwd 可能 全ユーザー読み取り可（644）
/etc/shadow 不可 rootのみ（640）

一括で制限する方法

  • Docker：プロジェクトディレクトリだけマウントし、~/.ssh や /etc はコンテナ内に存在させない
  • 専用ユーザー：Claude Code実行用ユーザーを作成し、機密ファイルへのアクセス権をOS側で剥奪
  • サンドボックスのdenyRead：sandbox.filesystem.denyRead で機密パスを列挙

おすすめ：サンドボックスを常時ONにしておくのがバランスが良い。一度設定すれば毎回の手間はない。

APIキーの安全な管理① — direnv

.env ファイルにAPIキーを直接書くのはNG。ディレクトリ単位で環境変数を自動管理しよう。

セットアップと使い方

# インストール
sudo apt-get install direnv

# シェルに連携を追加（~/.bashrc の末尾）
eval "$(direnv hook bash)"

# プロジェクトに .envrc を作成
echo 'export ANTHROPIC_API_KEY="sk-ant-..."' > .envrc

# 承認（変更のたびに必要）
direnv allow

# ディレクトリ移動で自動ロード/アンロード
cd your-project   # → 変数が自動ロード
cd ..             # → 変数が自動アンロード

セキュリティのポイント

  • .envrc は絶対にGitにコミットしない
  • グローバルな .gitignore で除外設定：
# グローバルgitignoreに追加
echo ".envrc" >> ~/.gitignore_global
git config --global core.excludesfile \
  ~/.gitignore_global

ディレクトリを抜けると変数は自動的に消える。別プロジェクトのClaude Codeセッションが機密情報を継承しない。

APIキーの安全な管理② — secret-tool

さらに安全に：APIキーをOSのキーリング（金庫）に保存し、ファイルには一切書かない。

保存と取得のコマンド

# APIキーをキーリングに保存（初回のみ）
secret-tool store \
  --label="Anthropic API Key" \
  service anthropic user developer
# → パスワード入力プロンプトが表示される

# .envrc から取得（direnvと組み合わせ）
export ANTHROPIC_API_KEY=$( \
  secret-tool lookup \
    service anthropic \
    user developer \
)

なぜこれが安全なのか

  • APIキーはメモリ上にのみ存在する
  • ディスク上のファイルに平文で書き込まれない
  • export コマンドを直接打たないのでシェル履歴に残らない
  • GNOMEキーリングが暗号化して管理

最強の組み合わせ：direnv + secret-tool
.envrc 内で secret-tool を呼び出せば、ファイルにもシェル履歴にもキーが残らない。

コミット前の最終防波堤 — Gitleaks

うっかりAPIキーをコミットしてしまうのを自動で防ぐ。

セットアップ

# インストール
sudo apt-get install gitleaks

# プリコミットフックとして設定
# .pre-commit-config.yaml に追加:
repos:
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.21.2
    hooks:
      - id: gitleaks

これで git commit のたびに自動スキャンが走り、APIキーのパターンが検出されるとコミットがブロックされる。

なぜ必要か

Gitは「追記専用」のデータモデル。一度コミットされたシークレットは、最新版から削除しても履歴に残り続ける。

  • Claude CodeはGit履歴を遡ってシークレットを発見しうる
  • 公開リポジトリなら外部の攻撃者にも見える
  • 履歴からの完全削除は非常に手間がかかる

「コミットさせない」のが最も確実な防御。

まとめ — 今日から実践する4つのこと

① アクセス制御を設定する

permissions.deny で機密ファイルをブロック +
サンドボックスを常時ONにしてBash経由のアクセスも防ぐ。

② 間接リークにも備える

Hooksで危険なコマンドを事前ブロック。CLAUDE.mdで出力ルールを定義。
機密性が高ければ実行は別ターミナルで。

③ APIキーを正しく管理する

.env に直接書かない。
direnv + secret-tool でファイルにもシェル履歴にも残さない。

④ Gitleaksを入れる

プリコミットフックでうっかりコミットを自動ブロック。
一度入った秘密は消せない。

Claude Codeは優秀だけど何でも読める実習生。
見せたくないものには鍵をかけよう。直接アクセスだけでなく出力経由のリークにも注意。
