Claude Code
スタートダッシュガイド

対象: Python経験のある新入社員 ／ 前提: Claude Code知識ゼロ

このガイドの思想

道具の性質を理解しないと効果が出ない。核となる考え方は3つ。

① セッションを使い捨てろ

コンテキストは有限。結論をファイルに書き出し、新セッションで次へ進む。

② ツールに任せろ

LLMは確率的。決定的な処理はスクリプトやリンターに委ねる。

③ コンテキストを守れ

探索はサブエージェントに委譲。必要な知識は必要なときだけ読み込む。

「必要なときだけ読み込む」＝段階的開示。
常駐知識はCLAUDE.md、タスク固有の知識は Skills で動的にロードする。

インストールと最初の起動

インストール

# インストール（Node.js 18以上が必要）
npm install -g @anthropic-ai/claude-code

# プロジェクトのルートに移動して起動
cd your-project
claude

初回起動時にログインするだけで準備完了。

確認コマンド

claude --version    # バージョン確認
claude --help       # ヘルプ表示

前提: Node.js 18以上
起動: プロジェクトルートで claude
終了: /exit または Ctrl+C

CLAUDE.md — プロジェクトの「取扱説明書」

セッション開始時に毎回読まれるMarkdown。LLM向けのプロジェクトREADME。

書くべきこと

  • ビルド・テスト・リントのコマンド
  • プロジェクト構造の概要
  • ツールで強制できない設計方針
  • 「やってはいけないこと」

書いてはいけないこと

  • ツールで強制できるルール（prettier、mypy等）
  • コードのコピペ（パスで参照せよ）
  • 詰め込みすぎ（多いと無視される）

CLAUDE.md — 良い例

# Project Overview
Python 3.12 / FastAPI / PostgreSQL のバックエンドAPI。

# Commands
- テスト: pytest tests/ -x --tb=short
- 型チェック: mypy src/ --strict
- リント: ruff check src/
- 単一テスト: pytest tests/test_xxx.py::test_name -v

# Architecture
- src/api/ — ルーティング
- src/domain/ — ビジネスロジック（外部依存なし）
- src/infra/ — DB・外部API

# Rules
- domain層はinfra層をimportしてはならない
- 新しいエンドポイントには必ず対応するテストを書く
- エラーハンドリングはsrc/api/errors.pyのパターンに従う

CLAUDE.md — 重要な原則と配置場所

重要な原則

  • 短く保て。 消してもClaudeが間違えないなら消す。
  • ポインタを使え。 コピペでなく src/api/errors.py を参照 と書く。
  • @構文で参照。 @README.md で外部ファイルを読み込める。
  • 定期的に手入れ。 指示無視が増えたら肥大化のサイン。

配置場所の使い分け

場所 用途
~/.claude/CLAUDE.md 全プロジェクト共通の個人設定
./CLAUDE.md プロジェクト固有（gitにcommitしてチームで共有）
./子ディレクトリ/CLAUDE.md サブモジュール固有（必要時のみ読み込まれる）

基本ワークフロー — セッションの使い捨て

最も大事な習慣: 1タスク1セッション。コンテキストが積むほど品質は落ちる。

推奨フロー

[セッション1: 相談]
  「このAPIの設計方針を相談したい」
  → 議論の結論をファイルに書き出させる
  → セッション終了

[セッション2: 実装]
  「docs/plan.md に書かれた方針に従って
   APIを実装して」
  → 新鮮なコンテキストで実装に集中
  → セッション終了

ストレージへの逃がし方

# セッション内でClaudeに指示する例
「今の議論の結論を
 docs/design-decision-auth.mdに書き出して」

「実装計画を
 docs/plan-user-api.mdにまとめて」

次のセッションではファイルを読むだけ。
コンテキストには結論だけが入る。

コンテキスト管理の小技

/compact

コンテキストを要約・圧縮する。

/clear

タスク切り替え時にコンテキストをリセット。

/model haiku

機械的な作業はHaikuで十分。速くて安い。

/effort low

深い推論が不要な作業に。速度UP・コスト削減。

LLMにやらせるな — スクリプト化と冪等性

原則: 決定的な処理はスクリプトに書け。 LLMは確率的。掛け算すら間違える。

悪い例

「このCSVの売上列を合計して」
→ LLMが自力で計算する
→ 間違える可能性あり

同じ入力でも毎回同じ結果とは限らない。

良い例

「このCSVの売上列を合計するPython
 スクリプトを書いて、
 scripts/sum_sales.py に保存して、
 実行して」
→ スクリプトが確定的に計算する
→ 再実行も可能

スクリプトは冪等。何度実行しても同じ結果。

品質管理もツールに委ねる

CLAUDE.mdに書いておくと効果的

- コード変更後は必ず
  `ruff check src/` と
  `mypy src/ --strict` を実行する
- テストは `pytest tests/ -x` で確認する
- これらが通らない状態で
  タスク完了とみなさない

さらに確実に — hooksで強制する

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "ruff check src/ && mypy src/ --strict"
          }
        ]
      }
    ]
  }
}

CLAUDE.mdの「お願い」 vs hooksの「強制」 — 確実なゲートにはhooksを使う。

mypyのもう一つの価値 — 型はLLMにとってのドキュメント

型アノテーション付きコード＝LLMにとってのコード兼ドキュメント。

NG: 型なし — LLMは推測するしかない

def calculate_discount(price, user, campaign):
    if campaign and user.is_premium:
        return price * campaign.rate
    return price

priceはint？float？推測は確率的で間違える。

OK: 型あり — シグネチャがドキュメント

from decimal import Decimal

def calculate_discount(
    price: Decimal,
    user: PremiumUser | StandardUser,
    campaign: DiscountCampaign | None,
) -> Decimal:
    if campaign is not None and isinstance(user, PremiumUser):
        return price * campaign.rate
    return price

型はmypyで検証される。常に正確なドキュメント。

サブエージェントへの委譲 — コンテキストを守る

探索・分析はコンテキストを大量消費する。メインを汚さずサブエージェントに委譲。

組み込みサブエージェント

Explore

読み取り専用のコードベース探索。

Plan

実装前の設計・計画立案。

汎用サブエージェント

探索＋変更の複合タスク用。結果だけ返る。

「Exploreサブエージェントを使って、認証関連のファイルを全部洗い出して」
「この変更の影響範囲をサブエージェントに調べさせて、結果だけ教えて」

カスタムサブエージェントの作り方

.claude/agents/ にMarkdownを配置:

<!-- .claude/agents/code-reviewer.md -->
---
name: code-reviewer
description: コード変更のレビュー。
  差分を読んで問題点を報告する。
tools:
  - Read
  - Grep
  - Glob
model: sonnet
---

あなたはコードレビュアーです。
与えられた差分を読み、以下の観点でレビュー:
- バグの可能性
- 設計原則への違反
- テストの漏れ

出力形式:
- 問題の重要度（高/中/低）
- ファイルパスと該当箇所
- 具体的な修正提案

ポイント

  • tools でツールを制限（最小権限）
  • model でコスト制御（sonnetで十分なことも）
  • ネスト不可（1階層のみ）
  • 結果だけ返る → コンテキスト節約

いつ委譲するか

状況 やること
「このプロジェクトの構造を把握したい」 Exploreに委譲
「この変更の影響範囲を調べたい」 サブエージェントに委譲
「テスト結果を分析してほしい」 サブエージェントに委譲
「このファイルを修正して」 メインで直接やる
「設計方針を相談したい」 メインで直接やる（その後セッション切り替え）

Skills — 段階的開示の中核

CLAUDE.md＝毎回読み込み。Skills＝必要時のみ読み込み。
この使い分けが段階的開示の実装。

skillの作り方

<!-- .claude/skills/deploy/SKILL.md -->
---
name: deploy
description: 本番環境へのデプロイ手順
---

# デプロイ手順

1. mainブランチが最新であることを確認
2. scripts/pre-deploy-check.sh を実行
3. scripts/deploy.sh production を実行
4. scripts/post-deploy-verify.sh でヘルスチェック

skillの設計方針

  • スクリプトやテンプレートを同梱可能
  • 「生成」でなく同梱ファイルを「使わせる」→ 再現性UP
  • 作り方はClaudeに聞ける: 「deploy用skillを作りたい」

常駐知識 → CLAUDE.md ／ タスク固有 → Skill

設定ファイルのベストプラクティス

.claude/settings.json の構成例

{
  "permissions": {
    "allow": [
      "Read",
      "Grep",
      "Glob",
      "pytest tests/**",
      "ruff check src/",
      "mypy src/ --strict"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "ruff check src/"
          }
        ]
      }
    ]
  }
}

設定の方針

  • permissions.allow で毎回の承認を省略
  • hooksでリンター・型チェックを自動実行
  • 「お願い」より「強制」（確定的 > 確率的）
  • gitにcommitしてチーム共有

/init で始める

cd your-project
claude
# セッション内で
/init

/init でCLAUDE.mdの雛形を自動生成。不要な記述を削って洗練させる。

カスタムコマンド — 繰り返し作業のパッケージ化

.claude/commands/ にMarkdownを置く → スラッシュコマンドになる。

<!-- .claude/commands/review-pr.md -->
---
description: PRの差分をレビューする
model: sonnet
---

git diffの結果をレビューしてください。
対象ブランチ: $ARGUMENTS

観点:
- バグの可能性
- テストの漏れ
- CLAUDE.mdのルールへの違反

問題がなければ「LGTM」とだけ返してください。

使い方

/review-pr feature/user-auth
  • $ARGUMENTS に引数が入る
  • チーム共有: .claude/commands/
  • 個人用: ~/.claude/commands/

コマンド化でチーム全員が同じ品質でタスクを実行。

よく使うコマンド一覧

コマンド 用途
/initCLAUDE.mdの雛形を生成
/compactコンテキストを要約・圧縮
/clearコンテキストをリセット
/model haiku軽量モデルに切り替え
/model opus高性能モデルに切り替え
/effort low推論の深さを下げる
/plan計画モード（実装前に計画を立てる）
/status現在の設定状態を確認
/doctor環境の問題を診断
/resume過去のセッションを再開
/hooks設定済みhookの一覧
/agentsサブエージェントの管理

ワンライナーでの利用

# パイプで渡す
cat error.log | claude -p "このエラーの原因を特定して"

# 非対話で実行
claude -p "tests/ 以下のテストを全部実行して結果を報告して"

-p フラグ: 非対話実行。CI/CDへの組み込みに便利。

まとめ — 明日から使う3つの習慣

① セッションを使い捨てろ

結論をファイルに書き出し、新セッションで実装。

② ツールに任せろ

hooksで自動化、計算はスクリプトで。

③ コンテキストを守れ

サブエージェントに委譲。Skillsで動的読み込み。

この3つでClaude Codeの効果は大きく変わる。
慣れたらカスタムコマンド、agent teams、git worktreeへ。
